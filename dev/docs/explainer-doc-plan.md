## 확정된 방향 (사용자 결정 v2)
1. **청중**: **Canton을 모르는 백엔드 개발자.** 블록체인 *기본 개념은 안다*(트랜잭션·블록·스마트컨트랙트·지갑·가스 등). → **이더리움 등 다른 네트워크와의 비교**를 적극 활용한다(개발자에게 가장 잘 통하는 설명축).
2. **위치/구성**: `wiki/` 안에 **신규 폴더** 1개 + **챕터(스테이지)별 별도 파일**. 단일 페이지 아님.
3. **형식**: 개념 설명서 + **"교육용"** — 이 문서로 *남에게 설명/강의*할 수 있어야 함. 각 챕터에 **강의 노트(어떻게 설명할지)** 포함. 
   - **학습 경험 레퍼런스**: bitcoin-study.vercel.app/learn?at=S0 … S13 — **스테이지를 클릭하며 한 단계씩**, 간단 실습·애니메이션, 인라인 설명. 이 "스테이지형 점진 학습" 느낌을 목표로 한다.
4. **기존 자료 재사용 금지**: `wiki/notes/*`를 링크로 엮지 않는다. 필요하면 **내용을 새 챕터 파일로 옮겨, 코스 흐름에 맞게 재작성**한다(소스로만 참고).
5. (유지) **대외비**: 외부 공유용 일반화 — 회사·Musubi·POC 참여사 실명 금지, 역할 기반 용어.

## 챕터별 핵심 (outline 초안 — 새 세션에서 확정받기)
- **S0 오리엔테이션**: "기관 간 정산을 블록체인으로? 그런데 왜 이더리움이 아니라 Canton?" 코스가 답할 3가지(프라이버시·원자성·신원/허가) + 코스 맵.
- **S1 문제**: 카운터파티 리스크("누가 먼저 보내?"), Herstatt 일화. 이더리움에서 escrow/atomic swap으로 풀 때의 한계(공개성·기관 부적합).
- **S2 이더리움 vs Canton 큰 그림**: account/EVM/글로벌 투명 상태/가스 vs 파티/Daml/이해관계자별 분산/트래픽. 비교표.
- **S3 데이터 모델**: 파티·참여자 노드·원장 — "노드는 자기 파티 컨트랙트만 저장". 이더리움 글로벌 복제와 대비.
- **S4 프라이버시**: 부분 트랜잭션 프라이버시(뷰 분해, 관련자만). 이더리움 투명성·메모리풀과 대비. (핵심 차별 ①)
- **S5 원자성·조합성**: 한 트랜잭션 전부/전무. EVM의 atomic composability와 비교 → Canton은 *프라이버시와 결합된* 원자성이 차별. (핵심 차별 ②)
- **S6 DvP 정산 흐름**: 제안→수락→개시→양측 잠금→원자적 실행 (시퀀스 다이어그램). 잠금(allocation)의 의미.
- **S7 토큰 출처**: 발행자=레지스트리(발행/소각·증빙). 외부발행+브릿지(래핑) vs Canton 직접발행 — 각 flow + 신뢰지점 비교.
- **S8 운영·배포**: 참여기관/운영사(venue)/발행자 역할. "정산 코드(DAR)는 모든 노드, 운영사 백엔드·venue 파티는 운영사만". 자체호스팅 vs NaaS.
- **S9 정리·실습·심화**: 한 장 요약(차별점 3) + 직접 해보기(인터랙티브 데모) + 다음에 읽을 것.
