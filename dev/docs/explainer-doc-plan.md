### 차용할 교육 패턴
- **하나의 시나리오를 끝까지**: "Alice→Bob 1 BTC"가 전 챕터를 관통 → 우리는 **"기관 A⇄B 통화 맞교환(FX DvP)" 한 건**을 관통시킨다.
- **질문으로 연다**: 각 스테이지를 "~는 어떻게 ~될까?" 질문으로 시작.
- **점진 빌드업**: 행동 → 소유권 → 규칙 → 저장 → 특성 → 운영.
- **프라이버시 위치 역전**: 비트코인은 S13(끝·각주). **우리는 프라이버시·원자성을 중앙(핵심 차별)으로.** 내러티브 훅으로 활용.
- **★ 구체성 + 기초/심화 계층** (S1 캡처에서 확인한 핵심): 추상 설명에서 멈추지 않고 **개발자가 바로 떠올리는 질문**("키는? 주소는? 코드는 어떻게 생겼나? 상태는 어디 저장되나?")에 **실제 값·구조·파생 과정**으로 답한다. 각 챕터를 `기초`(개념·비유·비교표) → `심화`(실제 식별자/스니펫/내부 동작)로 나눈다. bitcoin-study S1이 *주소→공개키→비밀키→ECDSA→해시 파생*까지 실값으로 보여준 깊이가 기준.
  - 도구: 친숙한 것→Canton 대응 **비교표**, 실제 문자열/스니펫(코드블록), 작은 워크드 예시, 인라인 다이어그램, 어려운 용어 툴팁, 🔷 이더리움 대비 박스.

## 관통 시나리오 (모든 챕터 공통)
> **기관 A가 통화 A를, 기관 B가 통화 B를 들고 있고, 둘을 맞바꾼다(FX DvP).** 운영사(venue)가 정산을 조율한다.
이 한 건의 정산이 소유권→규칙→저장→프라이버시→원자성→흐름→토큰→운영→확정성을 차례로 불러낸다.

- **index.md (코스 맵)**: 이 코스가 답하는 질문 · 대상(Canton 모르는 백엔드 개발자) · **스테이지 목록(S0~S11 링크)** · 읽는 법 · **이더리움→Canton 개념 대응표**(EOA/주소→파티, 솔리디티→Daml, 글로벌 공개상태→이해관계자별 분산, 투명→부분 프라이버시, 가스→트래픽, 확률적/에폭→즉시 결정적). 코스의 랜딩.
- **S0 오프닝**: "A가 B와 통화를 맞바꾸려 한다. 이 한 건이 무엇을 불러낼까?" 관통 시나리오를 세우고, 이 한 건이 불러낼 것들(소유권·규칙·저장·프라이버시·원자성·흐름·토큰·운영·확정성)을 예고 + "왜 이더리움이 아니라 Canton?" 떡밥. (코스 맵은 index.md로 분리)
- **S1 카운터파티 리스크**: "누가 먼저 보내?" 떼임 위험(Herstatt). 🔷 이더리움 atomic swap/escrow의 한계(공개·기관 부적합).
- **S2 파티 & 소유권**: "A가 이 자산을 쓸 수 있다는 건 어떻게 증명되나?" 파티·키·signatory/observer. 🔷 EOA/주소·서명.
- **S3 Daml 컨트랙트**: "정산 규칙은 어떻게 코드가 되나?" 템플릿·초이스·권한 모델. 🔷 솔리디티(누구나 호출) vs Daml(권한·이해관계자 명시).
- **S4 참여자 노드 & 원장**: "이 정산 기록은 어디에 저장되나?" 노드는 자기 파티 컨트랙트만. 🔷 이더리움 글로벌 복제 상태.
- **S5 프라이버시 ★차별①**: "A·B의 거래를 제3자가 볼까?" 부분 트랜잭션 프라이버시(뷰 분해). 🔷 이더리움 투명성·mempool. (훅: "비트코인은 이걸 맨 끝 각주로 뒀지만, 기관 정산에선 출발점")
- **S6 원자성 & DvP ★차별②**: "양쪽이 동시에, 아니면 둘 다 안 되게?" 한 트랜잭션 다리 묶기. 🔷 EVM atomic composability(되지만 공개) → Canton은 *프라이버시와 결합된* 원자성.
- **S7 정산 흐름**: "한 건이 실제로 어떻게 도나?" 제안→수락→개시→양측 잠금→원자적 실행 (시퀀스 다이어그램, 관통 시나리오로).
- **S8 토큰 & 레지스트리**: "맞바꾸는 그 통화는 어디서 왔나?" 발행자=레지스트리(발행/소각·증빙). 외부발행+브릿지(래핑) vs Canton 직접발행 — flow·신뢰지점.
- **S9 운영 & 배포**: "이걸 누가 운영하나?" 참여기관/운영사(venue)/발행자. "정산 코드(DAR)는 모든 노드, 운영사 백엔드·venue 파티는 운영사만". 자체호스팅 vs NaaS.
- **S10 확정성 & 합의**: "되돌릴 수 없나? 누가 순서를 정하나?" Synchronizer(시퀀서+미디에이터)·BFT·즉시 결정적 확정. 🔷 비트코인 확률적·이더리움 PoS finality.
- **S11 정리·실습·심화**: 차별점 3(프라이버시·원자성·신원/확정성) 요약 + 직접 해보기(인터랙티브 데모) + 더 배울 거리.

## 각 챕터의 "개발자 질문" + 구체 산출물 (S1 수준 구체성)
> 추상 금지. 각 챕터는 아래 *개발자가 떠올리는 질문*에 *실제 값/구조*로 답한다. (Canton 세부는 새 세션에서 demo·docs로 검증)

| 스테이지 | 개발자 질문 | 보여줄 구체 산출물 |
|---|---|---|
| S2 파티 | "키는? 주소는? 나는 어떻게 식별·서명되나?" | **파티 ID 구조** `app_user_…::1220abcd…`(사람용 hint + `::` + 네임스페이스 **키 지문**) · 서명 키 · **로컬 파티**(키를 참여자 노드가 보관) vs **외부 파티**(키를 파티가 직접 보관·제출 서명) · signatory/observer. 🔷 ETH EOA(주소=keccak(pubkey) 뒤 20B)와 대비 |
| S3 Daml | "솔리디티랑 뭐가 달라? 코드가 어떻게 생겼나?" | 실제 **template 스니펫**(signatory/observer/choice/controller) · 컨트랙트=활성 행, 소비형 choice=아카이브. 🔷 솔리디티(누구나 호출 가능) vs Daml(controller·이해관계자 명시) |
| S4 노드·원장 | "상태가 어디 저장돼? 글로벌이야? 내 노드엔 뭐가 있나?" | 참여자 노드 = 자기 파티가 stakeholder인 컨트랙트만(ACS) · Ledger API offset(증가하는 위치값). 🔷 ETH 전 노드 글로벌 상태 복제 |
| S5 프라이버시 | "tx가 mempool·익스플로러에 다 보이잖아. Canton은?" | 같은 트랜잭션을 파티별 토큰으로 조회 → **보이는 게 다름**(데모 수치 일반화: 당사자엔 보이고 제3자엔 0건) · 서브트랜잭션 뷰. 🔷 ETH 공개 mempool·익스플로러 |
| S6 원자성 | "여러 컨트랙트 한 tx에 묶기, EVM도 되잖아?" | 한 Daml 트랜잭션에 다중 exercise/create = 전부/전무 · 잠금(allocation). 🔷 EVM atomic composability(되지만 공개·가스 한도) |
| S7 정산 흐름 | "실제 호출 순서가 어떻게 되나?" | 실제 choice 시퀀스: 제안 create → `Accept` → `InitiateSettlement` → allocation → `Execute` (시퀀스 다이어그램) |
| S8 토큰 | "그 통화는 토큰으로 어떻게 표현돼?" | `instrumentId {admin, id}` · 레지스트리 오프레저 API(메타데이터·choice-context) · mint/burn |
| S9 운영 | "이걸 누가 어디에 배포하나?" | DAR vetting(모든 참여 노드) · 참여자 노드+Validator App 배포 · 백엔드는 각자. 자체호스팅 vs NaaS |
| S10 확정성 | "되돌릴 수 없나? 누가 순서를 정하나?" | Synchronizer(시퀀서+미디에이터) · 토폴로지/트랜잭션 서명 · BFT · 즉시 결정적 확정. 🔷 BTC 확률적·ETH PoS 에폭 finality |
