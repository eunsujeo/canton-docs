# 아키텍처 — B2B/B2C 역할 분담 & 멀티체인 (프로젝트)

> 우리 프로젝트의 계층 분담 결정 메모. 일반 Canton B2B/B2C 적합성은 위키 [Canton B2B vs B2C](../../wiki/notes/canton-b2b-vs-b2c.md) 참고.
> (dev 문서 — 위키와는 목적상 분리. 대외비는 아님: 공개 정보 기반.)

## 결정된 방향: 계층별 역할 분담
```
[B2C] 일반유저 ↔ 스테이블코인        → 퍼블릭 체인 (Base/Ethereum)
                       │ 브릿지(필수)
[B2B] 적격기관 ↔ 적격기관 정산(DvP)   → Canton (프라이버시·원자적 결정적 정산)
```
- **B2C는 Canton에서 안 한다.** 리테일 접점(소비자용 스테이블코인)은 **퍼블릭 체인**(Base/Ethereum)에서.
- **Canton은 B2B 정산 전용** — 기관 간 원자적 DvP(통화↔통화), 프라이버시 보존.
- 두 계층은 **브릿지로 연결**(필수 컴포넌트).

## 왜 이렇게 나누나
| 계층 | 체인 | 이유 |
|---|---|---|
| B2C(리테일) | 퍼블릭(Base/Eth) | 대량·저비용·간편 온보딩·기존 지갑 생태계(리테일이 이미 있음) |
| B2B(기관 정산) | Canton | 프라이버시(거래 상대·금액 비공개)·결정적 확정·허가형 자산·다자 권한 |

→ Canton 강점은 B2B에서만 결정적이고, B2C엔 오버스펙. 그래서 분담.

## DvP 정산이란 — 실제 교환 과정
아래 흐름도들은 정산을 **"Musubi DvP 정산" 상자 하나**로만 그리지만, 그 안에서 실제로 자산이 오가는 과정은 이렇다.

**정산(settlement)** = 거래 약속을 **실제 자산 이동으로 마무리**하는 단계.
문제는 *누가 먼저 보내나* — 국내은행이 KRW를 먼저 보냈는데 해외은행이 JPY를 안 보내면 떼인다(**카운터파티 리스크**).
**DvP(Delivery versus Payment)** 는 이걸 없앤다: **두 통화를 동시에, 한 묶음으로** 교환 → **전부 성공 or 전부 무효**, 한쪽만 가는 일은 구조적으로 불가능.

교환은 다음 순서로 일어난다(데모 구현과 동일):

```mermaid
sequenceDiagram
    autonumber
    participant A as 국내은행 (KRW 보유)
    participant V as Musubi (venue·운영자)
    participant B as 해외은행 (JPY 보유)
    A->>B: 거래 제안 (KRW ↔ JPY 맞교환)
    B-->>V: 수락
    V->>V: 정산 개시 (정산 계약 생성)
    A->>A: 내 KRW 잠금 (할당)
    B->>B: 내 JPY 잠금 (할당)
    V->>V: 정산 실행 — 단일 트랜잭션
    Note over A,B: KRW→해외은행, JPY→국내은행 동시 이동<br/>(둘 다 성공 또는 둘 다 무효 = 원자적)
```
1. **제안**: 국내은행이 "KRW ↔ JPY 맞교환"을 제안.
2. **수락**: 해외은행이 조건에 동의.
3. **개시**: venue(Musubi)가 정산을 개시 — 양 통화 다리를 묶은 *정산 계약* 생성.
4. **잠금(할당)**: 각 은행이 **자기 통화를 잠근다**(아직 안 넘어감, 예약만). 잠근 자산은 정산에만 쓸 수 있게 묶임.
5. **실행**: venue가 **단 하나의 트랜잭션**으로 두 통화를 **동시에** 맞바꿈. 한쪽이라도 부족·실패하면 **트랜잭션 전체가 무효** → 아무 자산도 안 움직임.

> 즉 "Musubi DvP 정산" 상자 = **③개시 → ④양측 잠금 → ⑤원자적 실행**. 이래서 **한쪽만 받고 떼이는 일이 불가능**(카운터파티 리스크 0)하고, 이게 Canton을 B2B 정산에 쓰는 핵심 이유다. (개념 더보기: 위키 [원자적 DvP](../../wiki/notes/atomic-dvp-real-differentiator.md))

## 미확정 (열어둘 것)
- **스테이블코인 발행 체인 세부**: KRWStable·JPYSC를 어느 체인에서 발행할지 **미정** → 아래 양쪽 시나리오 병기.
- 브릿지 방식(어떤 브릿지·신뢰 모델)도 설계 대상.

## 발행 시나리오 (양쪽 병기)
어느 쪽이든 **B2B 정산은 Canton(Musubi DvP)에서** 일어난다. 차이는 *정산에 쓰는 토큰이 Canton에 어떻게 들어오나*다.
정산이 도는 데 필요한 것: 그 토큰의 **레지스트리(발행/소각 관장 + 이체·정산 증빙 제공)** 가 Canton에 존재해야 함. (레지스트리=발행자가 운영, [stablecoin-instruments-plan](stablecoin-instruments-plan.md) 참고.)

### 시나리오 1 — 외부 체인 발행 + 브릿지
- 스테이블코인을 **퍼블릭 체인(Base/Ethereum)에서 발행**, Canton 정산용으로 **브릿지로 래핑(wrapped) 토큰** 반입.
- Canton의 **래핑 토큰 레지스트리 = 브릿지 운영자(또는 발행자 위임)** 가 운영. 정산은 이 래핑 토큰끼리 원자 교환.
- **신뢰 지점 = 브릿지**(락업 자산 ↔ 래핑 발행의 1:1 보장).

```mermaid
flowchart LR
  subgraph PUB["퍼블릭 체인 · Base/Ethereum"]
    ISS1["발행자"] --> BANKP["기관 지갑(퍼블릭)"]
  end
  subgraph BR["브릿지 (신뢰 지점)"]
    LOCK["락업/에스크로"]
  end
  subgraph CAN["Canton · B2B 정산"]
    REG1["래핑 토큰 레지스트리<br/>(브릿지 운영자)"]
    DVP1["Musubi DvP 정산"]
  end
  BANKP -->|"② 입금: lock"| LOCK
  ISS1 -.->|"① 발행 mint"| BANKP
  LOCK -->|"③ 래핑 mint"| REG1
  REG1 --> DVP1
  DVP1 -->|"④ 정산 후 출금: 래핑 burn"| LOCK
  LOCK -->|"⑤ unlock"| BANKP
```
1. 발행자가 퍼블릭 체인에서 KRWK/JPYSC 발행 → 기관 퍼블릭 지갑.
2. 기관이 정산에 쓰려고 브릿지에 입금(퍼블릭에서 **lock**).
3. 브릿지가 Canton에 **래핑 토큰 mint** → 기관 Canton 파티 보유.
4. Musubi DvP로 KRWK(wrapped) ↔ JPYSC(wrapped) **원자 정산**. 이후 기관이 출금하면 래핑 **burn**.
5. 브릿지가 퍼블릭에서 **unlock** → 기관 퍼블릭 지갑 복귀.

#### 따져보기 ⑴ — 프라이버시 경계: 무엇이 보이고 안 보이나
패턴 1엔 **원장이 둘**이고, 가시성이 다르다.

| | 퍼블릭 체인(Base/Eth) | Canton |
|---|---|---|
| 무엇이 일어나나 | 브릿지 입금(lock)·출금(unlock) | 실제 교환(DvP 정산) |
| 가시성 | **공개** — 누구나 봄 | **프라이빗** — 당사자 + venue만 |
| 드러나는 것 | 어느 주소가 Canton에 얼마 넣고/뺐나(순유입·순유출) | (외부엔 안 보임) |
| 가려지는 것 | — | 누가 누구와·얼마에·무슨 환율로 거래했나 |

- **Canton 안의 정산은 안 보인다**(부분 트랜잭션 프라이버시) — 거래 상대·금액·환율 비공개.
- **하지만 브릿지 입출금은 퍼블릭 체인에 그대로 찍힌다** — "A 주소가 100만 lock / 나중 98만 unlock"은 공개. **브릿지가 이걸 가리지는 못한다**(퍼블릭 체인은 본질적으로 투명).
- 즉 **가려지는 건 "정산 내역"이고, 브릿지 경계(온/오프램프)에서는 순유입·순유출이 샌다.** 관찰자는 *A가 Canton에 얼마 넣고 뺐는지*는 알지만, *그 안에서 누구와 무슨 거래를 했는지*는 모른다.

**완화책**: 네팅(개별 거래는 Canton에 두고 *순액만* 브릿지 통과) · 옴니버스/풀 주소(여러 기관 섞어 귀속 어렵게) · 배치. 그래도 퍼블릭 레그는 원천적으로 보인다.

#### 따져보기 ⑵ — burn↔unlock은 무슨 조건으로 일어나나
Canton과 퍼블릭 체인은 **서로를 native하게 모르는 별개 원장**이다. 그래서 "Canton에서 burn"과 "퍼블릭에서 unlock"은 자동 연결된 한 동작이 아니라 **서로 다른 두 이벤트**이고, 이 둘을 잇는 게 **브릿지**다.

> **unlock 조건 = 브릿지가 "*유효하고·최종이며·아직 상환 안 된* burn이 Canton에서 일어났다"고 확신**했을 때. 그 "확신"을 어떻게 만드냐 = **신뢰 모델**.

| 신뢰 모델 | unlock 조건(누가 확인) | 신뢰 가정 |
|---|---|---|
| **커스터디(중앙)** | 운영자가 Canton burn을 보고 퍼블릭에 unlock 서명 | 운영자 정직 (제일 약함) |
| **멀티시그/MPC(연합)** | M-of-N 검증자가 각자 burn 확인 후 공동 서명 | 과반 정직 |
| **라이트클라이언트/증명(탈중앙)** | 퍼블릭 컨트랙트가 *Canton burn 최종성 증명*을 직접 검증 | 암호학적(가장 강함·가장 어려움) |

**Canton 특유의 난점**:
- **burn이 프라이빗**이다. 브릿지가 확인하려면 **브릿지가 그 래핑 토큰/burn의 이해관계자(observer)** 여야 본다 → 래핑 토큰의 **레지스트리/admin = 브릿지 운영자**인 이유. 프라이빗 이벤트를 퍼블릭 체인에 누설 없이 증명하는 완전 trustless 방식은 매우 어려워, 현실은 대개 **운영자/연합 attestation**.
- **이중 unlock 방지**: 각 burn에 **고유 id(nonce)** → "한 burn = 한 unlock". 안 그러면 burn 없이 빼거나 한 번 burn하고 두 번 빼는 사고.
- **최종성**: burn이 **확정(되돌릴 수 없음)** 된 뒤 unlock. Canton은 **즉시 결정적 확정**이라 reorg 대기가 없는 게 유리.

#### 따져보기 ⑶ — 크로스체인은 원자적이 아니다 (가장 중요한 한계)
Canton **내부 DvP는 원자적**(전부/전무)이지만, **burn↔unlock은 두 체인에 걸친 별개 트랜잭션**이라 **원자성이 없다**. "burn은 됐는데 아직 unlock 안 된" 구간이 존재하고, 릴레이어/운영자가 죽으면 **자금이 묶일(liveness) 위험**이 있다. 보증·타임아웃·HTLC 같은 장치로 메우지만, 결국 **브릿지가 리스크·신뢰의 집결점**이다.

#### 따져보기 ⑷ — burn 뉘앙스
- **burn은 "퍼블릭으로 나갈 때(출금)"에만** 일어난다. 매 거래마다 X — 래핑 토큰은 Canton 안에서 **다음 정산에 계속 재사용**된다. 퍼블릭 체인으로 빼고 싶을 때만 burn.
- **burn하는 주체 = 그 시점 래핑 토큰 보유자.** 스왑 후엔 **상대방**일 수 있다(예: A가 wKRWK를 B에게 넘겼으면, 이제 **B가 burn → B의 퍼블릭 주소로 unlock**).

> ⇒ 패턴 1의 본질적 약점은 **브릿지(크로스체인 연결)** 다. **시나리오 2(Canton 직접 발행)** 는 이 연결 자체가 없어 더 단순·안전(대신 신뢰 지점이 발행자 준비금으로 이동). [아래 비교](#두-시나리오-비교) 참고.

### 시나리오 2 — Canton 직접 발행 (토큰표준)
- 스테이블코인을 **Canton 토큰표준으로 발행자가 직접 발행**. 브릿지 없이 네이티브.
- **통화마다 발행자(레지스트리)가 다르다** — KRWK는 KRW 발행자, JPYSC는 JPY 발행자가 각자 발행/소각·증빙. (한 발행자가 두 통화를 다 찍는 모델도 가능하나, KRW/JPY는 보통 각국 별도 발행자.)
- 신뢰 지점 = **각 발행자**(법정통화 준비금 ↔ 발행량). 퍼블릭/리테일(B2C)과 잇는다면 *그때* 반대 방향 브릿지 필요.

```mermaid
sequenceDiagram
    autonumber
    participant KI as KRW 발행자<br/>(KRWK 레지스트리)
    participant A as 국내은행
    participant V as Musubi (venue)
    participant B as 해외은행
    participant JI as JPY 발행자<br/>(JPYSC 레지스트리)
    KI->>A: KRWK 발행(mint)
    JI->>B: JPYSC 발행(mint)
    A->>B: 거래 제안 (KRWK ↔ JPYSC)
    B-->>V: 수락
    V->>V: 정산 개시 (정산 계약 생성)
    A->>A: KRWK 잠금(할당)
    B->>B: JPYSC 잠금(할당)
    V->>V: 정산 실행 — 단일 트랜잭션
    Note over A,B: KRWK→해외은행 · JPYSC→국내은행 동시 이동 (원자적)
    B->>KI: (선택) 받은 KRWK 상환 burn → 법정통화
    A->>JI: (선택) 받은 JPYSC 상환 burn → 법정통화
```
1. **각 통화 발행자**가 자기 통화를 해당 기관에 **발행(mint)** — KRW 발행자→국내은행(KRWK), JPY 발행자→해외은행(JPYSC).
2. 두 기관이 합의(제안·수락) → venue가 **정산 개시**.
3. 두 기관이 각자 통화 **잠금** → venue가 **단일 트랜잭션으로 동시 교환**(원자적, 브릿지 불필요).
4. (선택) 받은 통화를 **그 발행자에게 burn** → 법정통화 오프램프. (Canton 안에서 계속 재사용도 가능 — 매번 상환 X)

### 두 시나리오 비교
| 항목 | 시나리오 1 (외부발행+브릿지) | 시나리오 2 (Canton 직접발행) |
|---|---|---|
| 발행 장소 | 퍼블릭 체인(Base/Eth) | Canton(토큰표준) |
| 브릿지 | **필수** | 불필요(B2B 정산엔) |
| Canton 레지스트리 운영 | 브릿지 운영자(래핑) | 발행자 직접 |
| 신뢰 지점 | 브릿지(lock↔wrapped) | 발행자(준비금↔발행) |
| 리테일/B2C 연결 | 이미 퍼블릭에 있음(자연스러움) | 반대 방향 브릿지 필요 |
| 구현 부담 | 브릿지+래핑 레지스트리 | Canton 레지스트리만 |
| 데모(옵션 A) 매핑 | 래핑 토큰 레지스트리 구현 | 네이티브 토큰 레지스트리 구현 |

> 어느 쪽이든 **Musubi = venue(정산 운영자), 발행/소각 = 레지스트리(발행자)** 로 역할은 동일. 차이는 레지스트리를 *누가/어디서* 운영하느냐와 브릿지 유무.

## Musubi 매핑
- Musubi가 담당하는 건 **B2B 정산 계층(Canton)** — OTCTrade형 기관↔기관 DvP. (참고: [dvp-licensing-code-walkthrough.md](dvp-licensing-code-walkthrough.md))
- B2C 측(소비자 스테이블코인)은 Musubi/Canton 범위 밖 — 퍼블릭 체인 + 브릿지 연동 지점만 신경.
