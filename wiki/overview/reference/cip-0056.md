# CIP-0056: Canton Network 토큰 표준

[CIP-0056](https://github.com/global-synchronizer-foundation/cips/blob/main/cip-0056/cip-0056.md)은 Canton Network 토큰 표준을 정의한다 — 월렛과 애플리케이션이 레지스트리 전반의 <abbr class="gloss" title="실물·금융 자산을 원장 위의 토큰(컨트랙트)으로 표현하는 것">토큰화</abbr> 자산과 균일한 방식으로 상호작용하게 하는 Daml 인터페이스와 오프-<abbr class="gloss" title="거래·컨트랙트가 기록되는 장부. Canton에선 활성 컨트랙트의 모음">원장</abbr> API의 집합이다. CIP-0056을 구현한 레지스트리는 어떤 준수 월렛으로든 서비스될 수 있고, 어떤 애플리케이션이든 레지스트리별 코드에 의존하지 않고 이전과 정산을 조율할 수 있다.

<abbr class="gloss" title="글로벌 Synchronizer를 구동하는 오픈소스 애플리케이션 모음(SV·밸리데이터·월렛 등)">Splice</abbr> 월렛은 <abbr class="gloss" title="트랜잭션 수수료와 밸리데이터 보상에 쓰이는 네이티브 유틸리티 토큰(CC)">Canton Coin</abbr>(CC)에 대해 CIP-0056을 구현하며, 기반 Daml <abbr class="gloss" title="컨트랙트의 구조와 규칙(권한·초이스)을 정의하는 Daml 청사진">템플릿</abbr>은 `Amulet`이라 불린다. Canton Network의 다른 자산 레지스트리도 자신의 상품(instrument)에 대해 같은 인터페이스를 구현할 수 있다.

## CIP-0056이 표준화하는 것

표준은 세 범주의 기능을 다룬다:

* **포트폴리오 <abbr class="gloss" title="한 트랜잭션을 당사자별로 나눈 조각. 각 당사자는 자기 권한에 해당하는 뷰(자기 몫)만 받아 본다">뷰</abbr>** — 여러 레지스트리 전반의 <abbr class="gloss" title="Canton에서 권한과 데이터 가시성의 주체가 되는 식별 가능한 참여 주체">파티</abbr>의 현재 보유, <abbr class="gloss" title="원장 상태를 바꾸는 원자적 작업 단위. 하나 이상의 컨트랙트를 생성·보관하며, 전부 적용되거나 전혀 적용되지 않음">트랜잭션</abbr> 이력, 상품 메타데이터를 표시.
* **직접 P2P(FOP) 이전** — 자산을 한 파티에서 다른 파티로 이동, 단일 단계 또는 2단계 오퍼/수락 흐름으로.
* **인도-지불(DvP) 정산** — 다중 다리(multi-leg) 정산이 원자적으로 실행될 수 있도록 자산을 할당(allocation)에 잠금.

6개 온-원장 Daml API와 그에 대응하는 오프-원장(HTTP) 레지스트리 API가 이 역량을 지원한다:

* **Token Metadata** — 상품 심볼, 이름, 총 공급량, 레지스트리 UI URL
* **Holding** — 포트폴리오 뷰와 트랜잭션 이력
* **Transfer Instruction** — 직접 FOP 이전
* **Allocation** — DvP를 위한 자산 예약
* **Allocation Request** — 표준화된 앱→월렛 할당 요청
* **Allocation Instruction** — 표준화된 월렛→레지스트리 할당 생성

## 핵심 인터페이스

### Token Metadata

Token Metadata API는 월렛이 레지스트리의 내부 데이터 모델을 몰라도 상품에 대한 서술 정보를 가져오게 한다. 레지스트리는 상품의 표시 이름, 심볼, 총 공급량, 레지스트리 UI URL 같은 메타데이터를 노출한다. 월렛은 이를 레지스트리의 오프-원장 HTTP 엔드포인트로 조회한다.

### Holding

`Holding` 인터페이스는 단일 토큰 보유 — 파티 포트폴리오의 한 항목 — 를 나타낸다. 그 뷰는 다음을 노출한다:

* `owner` — 보유를 소유한 파티
* `instrumentId` — `admin` 파티(레지스트리)와 `id` 텍스트 문자열을 결합한 전역 고유 식별자
* `amount` — `Decimal`로 된 보유 크기
* `lock` — 보유자 파티, 만료 시간, 사람이 읽을 수 있는 컨텍스트 문자열을 가진 선택적 `Lock`
* `meta` — Kubernetes 어노테이션 규칙을 따르는 키-값 메타데이터

각 `Holding` <abbr class="gloss" title="원장에 기록되는 불변 데이터 단위. 상태 변경은 새 컨트랙트 생성으로 표현됨">컨트랙트</abbr>는 원장의 개별 UTXO다. `Holding` 인터페이스 ID(예: `#splice-api-token-holding-v1:Splice.Api.Token.HoldingV1:Holding`)로 필터링해 Ledger API의 <abbr class="gloss" title="아직 보관(소비)되지 않아 현재 유효한 컨트랙트">활성 컨트랙트</abbr> 엔드포인트로 보유를 조회한다.

### TransferFactory와 TransferInstruction

`TransferFactory`는 레지스트리가 유지하는 비소비형 팩토리 컨트랙트다. 월렛이 레지스트리의 오프-원장 API에서 그것을 가져와 `TransferFactory_Transfer` <abbr class="gloss" title="컨트랙트에서 수행 가능한 동작(권한이 부여된 당사자만 실행 가능)">초이스</abbr>를 실행해 이전을 개시한다. 이 초이스는 송신자, 수신자, 금액, 상품, 입력 보유, 데드라인(`executeBefore`)을 명시하는 `Transfer` 레코드를 받는다.

레지스트리 구현에 따라 결과는 다음 중 하나다:

* **Completed** — 이전이 즉시 정산되었고 수신자의 새 보유가 반환됨.
* **Pending** — `TransferInstruction` 컨트랙트가 생성되어 추가 동작이 필요함.
* **Failed** — 이전을 완료할 수 없어 입력 보유(수수료 제외)가 송신자에게 반환됨.

`TransferInstruction`이 생성되면, 진행 중인 이전의 상태를 추적한다. 수신자는 `TransferInstruction_Accept`나 `TransferInstruction_Reject`를 실행할 수 있고; 송신자는 `TransferInstruction_Withdraw`를 실행할 수 있다. 레지스트리는 `TransferInstruction_Update`로 내부 워크플로 단계를 진행할 수 있다.

### Allocation과 AllocationFactory

할당(Allocation)은 다중 다리 정산(DvP)을 지원한다. `Allocation` 컨트랙트는 잠긴 보유로 뒷받침되는, 정산의 특정 다리에 대한 자산 예약을 나타낸다.

`AllocationView`는 다음을 담는다:

* `allocation` — 정산 정보, 이전 다리 ID, 이전 다리 자체(송신자, 수신자, 금액, 상품)를 참조하는 `AllocationSpecification`
* `holdingCids` — 이 할당을 뒷받침하는 보유
* `meta` — 확장성 메타데이터

레지스트리에서 `AllocationFactory`를 가져와 `AllocationFactory_Allocate`를 실행해 할당을 생성한다. 할당 컨트랙트를 발견·상호작용하려면 [토큰 표준 API](splice-wallet-reference.md#월렛-api-엔드포인트)를 쓴다. 정산의 `executor` 파티가 송신자·수신자와 함께 할당을 공동 제어한다. 세 초이스가 그 생애주기를 좌우한다:

* `Allocation_ExecuteTransfer` — 할당된 자산의 이전 실행(정산 단계)
* `Allocation_Cancel` — 송신자·수신자·executor가 공동 제어하는, 할당의 조기 해제
* `Allocation_Withdraw` — 송신자가 `allocateBefore` 데드라인 전에 자산 회수

`AllocationInstruction` 인터페이스는 `TransferInstruction`이 다단계 이전을 추적하는 것과 유사하게 다단계 할당 생성을 추적한다.

### Allocation Request

`AllocationRequest` 인터페이스는 애플리케이션이 월렛에 자산 할당을 요청하는 방식을 표준화한다. 앱이 정산 상세와 이전 다리를 명시하는 할당 요청을 생성하면; 사용자의 월렛이 적절한 `AllocationFactory` 초이스를 실행해 자금을 댄다. 이는 앱의 정산 로직을 월렛의 보유 관리와 분리한다.

## 이전 모델

### 1단계 이전

1단계 이전은 단일 Daml 트랜잭션 내에서 완료된다. `TransferFactory_Transfer`를 실행하고 레지스트리가 즉시 정산할 수 있으면, 결과는 수신자의 새 보유와 함께 `TransferInstructionResult_Completed`다.

Canton Coin의 경우, 수신자가 `TransferPreapproval` 컨트랙트를 설정했을 때 1단계 이전이 작동한다. 사전 승인은 수신자가 이전당 동의 없이 어떤 송신자로부터든 들어오는 CC를 받기로 동의함을 신호한다. 각 사전 승인은 `receiver` 파티와 `provider` 파티를 가지며; provider는 사전 승인 수명에 비례한 수수료를 내고 그것을 쓰는 들어오는 이전에 대해 앱 보상을 번다. 사전 승인은 만료되어 주기적으로 갱신되어야 한다 — provider가 <abbr class="gloss" title="파티를 호스팅하고 그 파티의 컨트랙트 데이터를 저장하는 참여자 노드">밸리데이터</abbr> 운영자면 밸리데이터 앱이 이를 자동 처리한다.

Splice 월렛과 토큰 표준 레지스트리 API는 기존 사전 승인을 감지해 이전을 그것을 통해 자동 라우팅한다. 다른 레지스트리는 자체 1단계 이전 메커니즘을 구현하거나 항상 2단계 흐름을 요구할 수 있다.

### 2단계 이전

수신자가 사전 승인이 없거나, 레지스트리가 추가 권한을 요구할 때, 이전은 두 단계를 거친다:

1. **Instruct(지시)** — 송신자가 `TransferFactory_Transfer`를 실행한다. 레지스트리가 `TransferPendingReceiverAcceptance` 상태의 `TransferInstruction` 컨트랙트를 생성한다. 송신자의 입력 보유가 소비되고, 자금은 레지스트리가 보유한다.
2. **Accept(수락)** — 수신자가 대기 중인 지시에 `TransferInstruction_Accept`를 실행한다. 레지스트리가 이전을 완료하고 수신자를 위한 보유를 생성한다.

수신자는 지시를 거부(`TransferInstruction_Reject`)할 수도 있으며, 이는 자금을 송신자에게 반환한다. 송신자는 수락 전에 철회(`TransferInstruction_Withdraw`)할 수 있다. 두 경로 모두 월렛이 트랜잭션 이력을 파싱해 주어진 지시가 성공했는지 실패했는지 판정할 수 있도록 설계되었다.

2단계 이전은 초기 지시 후 송신자 측 자동화가 필요 없어, 자신이 통제하는 키로 트랜잭션에 서명하는 외부 파티에 잘 맞는다. [CIP-0078](https://github.com/global-synchronizer-foundation/cips/blob/main/cip-0078/cip-0078.md) 이후 2단계 CC 이전은 수수료가 없다.

### 할당을 쓰는 DvP 정산

정산이 서로 다른 상품이나 레지스트리에 걸친 여러 다리를 포함할 때, 각 송신자가 독립적으로 자산을 할당한다:

1. 애플리케이션이 `executor` 파티, 데드라인(`allocateBefore`), 정산 데드라인(`settleBefore`)을 가진 정산 명세를 생성한다.
2. 각 송신자의 월렛이 `AllocationFactory_Allocate`를 실행해 각자의 이전 다리에 대한 `Allocation` 컨트랙트에 보유를 잠근다.
3. 모든 다리가 할당되면, executor가 각 할당에 `Allocation_ExecuteTransfer`를 실행해 정산을 원자적으로 완료한다.

송신자가 데드라인 전에 할당하지 못하면, executor가 남은 할당을 취소할 수 있다. 송신자는 `allocateBefore` 시각 전에 할당을 철회해 보유를 회수할 수도 있다.

## UTXO 관리

Bitcoin과 유사하게, Canton은 UTXO 모델을 쓰며, 여기서 UTXO는 `Holding` 인터페이스를 구현하는 모든 활성 컨트랙트다.

활성 `Holding` 컨트랙트는 `Holding` 컨트랙트를 소유한 사용자를 호스팅하는 밸리데이터 노드와 토큰 관리자를 호스팅하는 밸리데이터 노드 모두에 저장·연산 비용을 발생시킨다. 네트워크 자원을 효율적으로 쓰려면, 월렛 제공자는 사용자당 UTXO 수를 낮게 — 사용자당 평균 대략 10개 미만으로 — 유지하는 것을 목표로 해야 한다.

이는 또한 <abbr class="gloss" title="Synchronizer에 쓰기를 요청할 때 소비하는 자원. Canton Coin으로 비용을 지불">트래픽</abbr> 비용을 최적화한다. 이전에 입력되는 각 UTXO가 추가 트래픽 비용을 들기 때문이다. 나아가 이 권고는 Canton Coin 같은 토큰이 마련한 인센티브와 정렬된다. Canton Coin은:

* 단일 이전에 최대 100개의 입력 컨트랙트만 허용해 보유의 과도한 분할을 막고,
* 초기 금액이 누적 보유 수수료보다 낮은 코인 UTXO를 만료시켜 더스트 UTXO 생성을 막는다.

월렛 제공자는 이전 입력으로 소액 `Holding` UTXO를 우선 선택하고, 사용자 온보딩 중 `MergeDelegation` 컨트랙트를 설정해 월렛이 사용자를 대신해 작은 UTXO를 자동 병합할 수 있게 해야 한다. Splice `splice-util-token-standard-wallet.dar` 패키지가 이 목적의 `MergeDelegation`·`BatchMergeUtility` 템플릿을 제공한다.

## 레지스트리와 발견

각 토큰 상품은 레지스트리 — 소유 기록을 유지하고 온-원장 Daml 인터페이스와 오프-원장 HTTP API를 모두 노출하는 애플리케이션 — 가 관리한다. 레지스트리의 `admin` 파티는 모든 `InstrumentId`와 모든 팩토리 컨트랙트의 뷰에 나타난다.

팩토리 초이스를 실행하려면, 먼저 레지스트리의 오프-원장 API를 조회해 다음을 얻는다:

* `factoryId`(팩토리의 컨트랙트 ID)
* `disclosedContracts`(실행 성공에 필요)
* `choiceContextData`(초이스 인자에 `context`로 전달)

주어진 상품의 레지스트리 URL은 현재 월렛 자체가 유지한다. [Canton 네임 서비스](canton-name-service.md) 기반의 일반 발견 메커니즘이 계획되어 있으며, 레지스트리 URL이 CNS 엔트리 메타데이터에 저장되어 Scan API로 가져올 수 있게 될 것이다.

오프-원장 API는 인증을 요구하지 않는다. 대신 보안 모델은 컨트랙트 ID의 추측 불가능성과 Canton의 알 필요(need-to-know) 데이터 분산에 의존한다 — 컨트랙트를 볼 권한이 있는 파티만 그것을 받는다.

## 메타데이터 규칙

모든 CIP-0056 인터페이스는 `Metadata` 필드 — 확장성을 위한 키-값 텍스트 맵 — 를 포함한다. 키는 Kubernetes 어노테이션 규칙을 따르며 키를 정의하는 조직의 DNS 이름을 접두사로 붙여야 한다. 예컨대 Splice 구현은 `splice.lfdecentralizedtrust.org/` 접두사 키를 쓴다.

트랜잭션 파싱에 쓰이는 표준 메타데이터 키:

* `splice.lfdecentralizedtrust.org/tx-kind` — 연산 유형(예: `transfer`, `merge-split`, `burn`, `mint`, `unlock`, `expire-dust`)
* `splice.lfdecentralizedtrust.org/sender` — 보내는 파티
* `splice.lfdecentralizedtrust.org/reason` — 연산의 사람이 읽을 수 있는 사유
* `splice.lfdecentralizedtrust.org/burned` — 소각된 보유의 양

`ExtraArgs` 타입은 대부분의 초이스에 전달되는 두 구조를 묶는다: `ChoiceContext`(레지스트리의 오프-원장 API에서 온 불투명 데이터)와 `Metadata`(호출자가 제공하는 어노테이션). 초이스 컨텍스트는 레지스트리 내부 데이터를 온-원장 실행에 전달하고; 메타데이터는 호출자와 레지스트리가 추가 의미론에 <abbr class="gloss" title="여러 노드가 트랜잭션의 유효성·순서에 함께 동의하는 절차">합의</abbr>하게 한다.

## 월렛 통합

[Wallet SDK](https://github.com/canton-network/wallet-gateway/tree/main/sdk/wallet-sdk)는 이 통합 패턴을 월렛 개발자를 위한 재사용 가능 라이브러리로 패키징한다. 월렛을 처음부터 구축한다면, 다섯 통합 패턴은:

* 토큰 표준 인터페이스를 구현하는 컨트랙트 읽기(활성 컨트랙트 엔드포인트로)
* 토큰 표준 컨트랙트가 관여하는 트랜잭션 이력 읽기·파싱
* 팩토리 초이스 실행(레지스트리에서 팩토리를 가져온 뒤 Ledger API에 제출)
* 기존 컨트랙트의 비팩토리 초이스 실행(예: 이전 지시 수락)
* 커스텀 Daml 코드에서 토큰 표준 초이스 호출(대량 연산, 병합, 앱별 워크플로용)

[토큰 표준 CLI](https://github.com/canton-network/splice/tree/main/token-standard#cli)는 이 패턴 각각을 실행 가능한 코드로 보여주며, 테스트 도구이자 레퍼런스 구현 역할을 한다.
