# 트랜잭션 생애주기

> Canton 트랜잭션의 완전한 5단계 생애주기 — 준비부터 시퀀싱, 검증, 최종 커밋까지

Canton 트랜잭션은 커밋(또는 거부)되기 전에 다섯 개의 뚜렷한 단계를 거친다. 각 단계는 특정 노드 역할 — 참여자 노드, 시퀀서 노드, 미디에이터 노드 — 이 암호화된·시퀀싱된 메시지를 통해 통신한다. 어떤 단일 노드도 전체 트랜잭션을 평문으로 보지 못한다.

단계는 다음과 같다:

1. **준비(Preparation)** — 제출 참여자가 커맨드를 해석하고 암호화된 트랜잭션 뷰를 구성
2. **제출(Submission)** — 참여자가 확인 요청 배치를 시퀀서에 보냄
3. **시퀀싱·분배(Sequencing and Distribution)** — 시퀀서가 메시지를 순서화하고 수신자에게 전달
4. **검증·확인(Validation and Confirmation)** — 각 확인 참여자가 자기 뷰를 검증하고 미디에이터에 응답
5. **집계·커밋(Aggregation and Commit)** — 미디에이터가 응답을 수집해 평결을 내리고, 참여자가 평결에 따라 결과를 적용(또는 폐기)

## 1단계: 준비

생애주기는 제출 참여자 노드가 Ledger API(gRPC)를 통해 애플리케이션으로부터 커맨드를 받으면서 시작된다. 그러면 참여자는:

* <abbr class="gloss" title="다자간 워크플로를 위해 설계된 Canton의 스마트 컨트랙트 언어">Daml</abbr> 엔진에서 Daml 스마트 <abbr class="gloss" title="원장에 기록되는 불변 데이터 단위. 상태 변경은 새 컨트랙트 생성으로 표현됨">컨트랙트</abbr> 코드를 로컬로 실행해 **커맨드를 해석한다**. 엔진은 참여자의 현재 활성 컨트랙트 집합(ACS)에 대해 커맨드를 평가하고 전체 트랜잭션 트리 — 모든 액션(생성·실행·페치), 각 액션의 결과, 모든 관련 메타데이터를 담은 구조 — 를 생성한다.

* 트리의 각 액션 노드에 대해 **인포미(informees)를 결정한다**. 인포미는 Daml <abbr class="gloss" title="컨트랙트의 구조와 규칙(권한·초이스)을 정의하는 Daml 청사진">템플릿</abbr>이 정의한 대로 주어진 액션을 볼 권한이 있는 <abbr class="gloss" title="Canton에서 권한과 데이터 가시성의 주체가 되는 식별 가능한 참여 주체">파티</abbr> — <abbr class="gloss" title="컨트랙트의 주된 권한자. 생성·보관(소비)에 반드시 동의해야 하는 파티">서명자</abbr>, <abbr class="gloss" title="컨트랙트를 볼 수 있으나 단독으로 행위할 수는 없는 파티">관찰자</abbr>, 액터(<abbr class="gloss" title="컨트랙트에서 수행 가능한 동작(권한이 부여된 당사자만 실행 가능)">초이스</abbr> 실행자) — 다. 각 액션의 인포미 집합이 누가 트랜잭션의 어떤 부분을 받을지 좌우한다.

* **트랜잭션을 뷰로 분해한다.** 트랜잭션 트리의 각 액션 노드가 별도 뷰가 된다. 뷰는 액션, 그 결과, 검증에 충분한 컨텍스트를 담지만 그 이상은 아니다. 각 뷰는 그 인포미(와 그 인포미를 호스팅하는 참여자)만 복호화할 수 있도록 암호화된다. 비인포미는 액션에 관해 아무것도 알지 못한다: 페이로드도, 파티도, 액션의 존재조차도.

* 전체 트랜잭션 트리에 암호학적으로 커밋하는 **루트 해시(root hash)를 계산한다**. 루트 해시는 모든 뷰를 함께 묶는다: 모든 확인 참여자가 자신이 받은 뷰가 다른 모든 참여자가 검증하는 것과 같은 트랜잭션에 속함을 검증할 수 있다. 미디에이터는 루트 해시를 써서 확인 응답을 상관짓는다.

트랜잭션이 여러 <abbr class="gloss" title="상태를 저장하지 않고 트랜잭션 합의·순서를 조율하는 Canton 구성요소">Synchronizer</abbr>의 컨트랙트를 건드리면, 참여자는 제출 전에 그 컨트랙트를 공통 Synchronizer로 <abbr class="gloss" title="컨트랙트를 한 Synchronizer에서 다른 Synchronizer로 옮기는 프로토콜">재할당</abbr>해야 할 수 있다. 이는 [재할당 프로토콜](reassignment-protocol.md)이 처리한다.

## 2단계: 제출 (확인 요청)

제출 참여자는 시퀀서에 **확인 요청 배치(confirmation request batch)** 를 보낸다. 배치는 세 범주의 메시지를 담는다:

* **암호화된 뷰** — 액션 노드당 하나. 각 뷰는 그 인포미를 호스팅하는 참여자를 위해 암호화된다. 특정 뷰의 어떤 인포미도 호스팅하지 않는 참여자는 그것을 복호화할 수 없고 결코 받지 않는다.

* **루트 해시 메시지** — 수신 참여자당 하나. 각 루트 해시 메시지는 트랜잭션의 루트 해시를 담아, 수신자가 받은 뷰를 단일 트랜잭션에 상관지을 수 있게 한다.

* **인포미 메시지** — 미디에이터에게 전송. 각 뷰의 인포미와 확인 정책을 나열한다. 확인 정책은 어떤 서명자가 확인해야 하는지와, 각 서명자에 대해 승인해야 하는 호스팅 참여자의 임계값을 명시한다. 미디에이터는 이 정보를 써서 충분한 확인이 도착했는지 판정한다.

제출 참여자는 이 모든 메시지를 단일 배치로 시퀀서에 보낸다.

## 3단계: 시퀀싱·분배

시퀀서는 확인 요청 배치를 처리한다:

1. **타임스탬프를 부여한다.** 이것이 트랜잭션의 시퀀싱 시간이다. 시퀀서는 BFT-쓰기를 수행한다: 순서화 노드가 전역 메시지 시퀀스에서 이 배치의 위치와 타임스탬프에 합의한다. 이 타임스탬프가 이후 모든 데드라인의 기준점이 된다.

2. **각 메시지를 지정된 수신자에게만 전달한다.** 참여자는 자신이 인포미 파티를 호스팅하는 암호화된 뷰와 자신의 루트 해시 메시지를 받는다. 미디에이터는 인포미 메시지를 받는다. 트랜잭션의 어떤 뷰에 대해서도 인포미를 호스팅하지 않는 참여자는 아무것도 받지 않고 트랜잭션에 관해 아무것도 알지 못한다.

미디에이터는 인포미 메시지를 받으면:

* 확인 요청 메타데이터(루트 해시, 인포미 목록, 확인 정책)를 기록한다
* 시퀀싱 시간과 Synchronizer의 구성된 결정 시간 파라미터를 기반으로 **확인 데드라인 타이머**를 시작한다
* 참여자로부터의 확인 응답을 기다린다

## 4단계: 검증·확인

뷰를 받은 각 참여자는 독립적으로 그것을 검증한다. 검증 검사는:

* **적합성(Conformance)** — 참여자가 받은 뷰의 각 액션에 대해 Daml 컨트랙트 로직을 재실행한다. 재실행은 뷰에서 주장된 것과 같은 결과(생성된 컨트랙트, 보관된 컨트랙트)를 내야 한다. 결과가 다르면, 뷰가 잘못되었거나 제출자가 부정직한 것이다.

* **권한(Authorization)** — 참여자가 모든 액션이 올바르게 인가되었는지 확인한다. 각 액션에 대해 필요한 파티(서명자, 컨트롤러)가 인가했어야 한다. 권한은 파티-참여자 <abbr class="gloss" title="어떤 노드·파티·키가 네트워크에 참여하는지를 정의하는 구성 정보">토폴로지</abbr>와 Daml 권한 규칙으로 검증된다.

* **일관성(Consistency)** — 참여자가 뷰가 참조하는 모든 입력 컨트랙트가 자기 로컬 ACS에서 여전히 활성인지 확인한다. 이미 소비된(보관된) 컨트랙트는 다시 쓸 수 없다. 이 검사가 단일 Synchronizer 내 이중지불을 막는다.

* **시간 경계(Time bounds)** — 참여자가 트랜잭션에 주장된 원장 시간이 시퀀싱 시간(기록 시간)에 대해 허용 가능한 경계 내에 드는지 검증한다. 허용 창은 Synchronizer 파라미터다.

* **루트 해시(Root hash)** — 참여자가 받은 뷰가 루트 해시 메시지의 루트 해시와 일관되는지 검증한다. 이는 모든 참여자가 같은 트랜잭션 트리의 뷰를 검증하도록 보장해, 악의적 제출자가 다른 참여자에게 다른 뷰를 보내는 것을 막는다.

검증 후, 참여자는 **미디에이터 응답(mediator response, MR)** 메시지를 구성한다:

* 참여자가 받은 모든 뷰에 대해 모든 검사가 통과하면 **승인(Approve)**
* 어떤 검사든 실패하면 거부 사유와 함께 **거부(Reject)**

미디에이터 응답은 시퀀서로 전송되며, 시퀀서가 또 다른 BFT-쓰기로 메시지에 타임스탬프를 찍은 뒤 미디에이터에 전달한다. 확인 참여자는 응답을 미디에이터에 직접 보내지 않는다 — 모든 통신은 시퀀서를 통해 흐른다.

## 5단계: 집계·커밋

미디에이터는 미디에이터 응답 메시지를 수집해 확인 정책에 대해 평가한다:

1. **집계(Aggregation).** 각 뷰에 대해 확인 정책은 어떤 서명자 파티가 확인해야 하는지 명시한다. 서명자 파티는 여러 참여자에 호스팅될 수 있다; 정책은 확인 임계값 — 그 서명자가 확인된 것으로 간주되려면 몇 개의 호스팅 참여자가 승인해야 하는지 — 을 정의한다. 미디에이터는 서명자별·뷰별로 승인을 추적한다. 여기에는 제출 파티가 다중 호스팅인 경우도 포함된다.

2. **평결 판정.** 두 결과가 가능하다:
   * **승인**: 확인 데드라인 전에 모든 뷰의 모든 서명자 임계값이 충족됨. 모든 필요한 서명자에 대해 충분한 호스팅 참여자가 승인.
   * **거부**: 필요한 참여자 중 하나라도 명시적으로 거부하거나, 충분한 승인 없이 확인 데드라인이 만료됨. 단일 서명자가 임계값을 충족하지 못하면 전체 트랜잭션을 거부하기에 충분하다.

3. **트랜잭션 결과 분배.** 미디에이터는 시퀀서에 **트랜잭션 결과(transaction result, TR)** 메시지를 보낸다. 시퀀서는 이 평결을 모든 인포미 참여자에게 전달한다. 참여자가 시퀀서로부터 평결을 읽을 때 BFT-읽기를 수행한다 — 전달된 메시지의 시퀀서 서명을 검증해 진위를 확인한다.

4. **로컬 커밋 또는 폐기.** 각 참여자가 평결을 처리한다:
   * **승인 시**: 참여자가 트랜잭션을 커밋한다. 소비된 컨트랙트를 보관하고, 새 컨트랙트를 생성하고, ACS를 갱신한다. 트랜잭션과 그 효과가 참여자의 로컬 원장 투영의 일부가 된다.
   * **거부 시**: 참여자가 트랜잭션을 폐기한다. 어떤 상태 변경도 적용되지 않는다. ACS는 변하지 않는다.

평결이 시퀀싱되어 전달되면, 트랜잭션 결과는 **최종**이다. 포크, 재조직, 롤백이 없다. 커밋된 트랜잭션은 커밋된 채로, 거부된 트랜잭션은 거부된 채로 남는다.

## 메시지 요약

다음 표는 프로토콜 중 교환되는 메시지, 발신자, 수신자, 내용을 요약한다.

| 메시지 | 발신자 | 수신자 | 내용 |
| --- | --- | --- | --- |
| **커맨드(Command)** | 애플리케이션 | 제출 참여자 | Ledger API를 통한 Daml 커맨드(생성, 실행) |
| **암호화된 뷰** | 제출 참여자 | 시퀀서 → 인포미 참여자 | 액션 노드당 하나; 인포미별 암호화 |
| **루트 해시 메시지** | 제출 참여자 | 시퀀서 → 각 수신 참여자 | 전체 트랜잭션 트리의 루트 해시 |
| **인포미 메시지** | 제출 참여자 | 시퀀서 → 미디에이터 | 뷰별 인포미 목록과 확인 정책 |
| **미디에이터 응답(MR)** | 확인 참여자 | 시퀀서 → 미디에이터 | 승인 또는 거부(사유 포함) |
| **트랜잭션 결과(TR)** | 미디에이터 | 시퀀서 → 모든 인포미 참여자 | 최종 평결: 승인 또는 거부 |

## 타이밍과 데드라인

두 시간 파라미터가 프로토콜의 라이브니스를 좌우한다:

* **시퀀싱 시간(Sequencing time)** — 확인 요청 배치가 순서화될 때 시퀀서가 부여. 트랜잭션의 기준 타임스탬프다.
* **결정 시간(Decision time)** — 시퀀싱 시간에 Synchronizer의 `decisionTimeout` 파라미터를 더해 계산. 미디에이터가 결정 시간까지 충분한 확인을 받지 못하면, 트랜잭션은 타임아웃으로 거부된다.

참여자는 또한 **참여자 응답 데드라인**을 강제한다: 시퀀싱 시간에서 파생된 구성된 컷오프 전에 미디에이터 응답을 보내야 한다. 제때 응답하지 못한 참여자는 그 트랜잭션에 대해 비확인으로 취급된다.

## 실패 모드

트랜잭션은 생애주기의 여러 지점에서 실패할 수 있다:

* **준비 실패**: Daml 엔진이 커맨드를 거부(예: 컨트랙트 미발견, 권한 실패, 단언 위반). 어떤 메시지도 보내지기 전에 커맨드가 동기적으로 거부된다. 다른 노드는 관여하지 않는다.

* **검증 거부**: 확인 참여자가 적합성·권한·일관성·시간 경계 위반을 발견. 거부 MR을 보낸다. 확인 정책에 따라 단일 거부가 전체 트랜잭션 거부에 충분할 수 있다.

* **타임아웃**: 결정 시간 전에 충분한 참여자가 응답하지 않음. 미디에이터가 트랜잭션을 거부한다. 참여자가 오프라인이거나 과부하거나 도달 불가일 때 일어날 수 있다.

* **이중발화(equivocation) 탐지**: 제출자가 다른 참여자에게 일관되지 않은 뷰(루트 해시와 맞지 않는 뷰)를 보내면, 정직한 참여자가 루트 해시 검증 중 불일치를 탐지하고 거부한다.

모든 실패 경우에 어떤 상태도 바뀌지 않는다. 모든 참여자의 ACS는 트랜잭션 제출 전 그대로 남는다.
