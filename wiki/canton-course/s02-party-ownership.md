# S2 — 파티 & 소유권

## 질문
**A가 이 자산을 쓸 수 있다는 건 어떻게 증명되나? 키는 무엇이고, 주소는? 나는 어떻게 식별·서명되나?**

## 기초

이더리움에서 "나"는 **주소**다. 개인키로 주소를 파생하고, <abbr class="gloss" title="원장 상태를 바꾸는 원자적 작업 단위. 하나 이상의 컨트랙트를 생성·보관하며, 전부 적용되거나 전혀 적용되지 않음">트랜잭션</abbr>에 서명하면 그 주소가 행위자가 된다. Canton에서 "나"는 **<abbr class="gloss" title="Canton에서 권한과 데이터 가시성의 주체가 되는 식별 가능한 참여 주체">파티</abbr>(party)**다. 비슷해 보이지만 결정적 차이가 있다.

### 파티란?
파티는 Canton에서 **권한과 데이터 가시성의 주체**다. "국내 기관 A", "해외 기관 B"가 각각 하나의 파티다. <abbr class="gloss" title="원장에 기록되는 불변 데이터 단위. 상태 변경은 새 컨트랙트 생성으로 표현됨">컨트랙트</abbr>(다음 단계)는 어떤 파티가 **<abbr class="gloss" title="컨트랙트의 주된 권한자. 생성·보관(소비)에 반드시 동의해야 하는 파티">서명자</abbr>(signatory)**이고 어떤 파티가 **<abbr class="gloss" title="컨트랙트를 볼 수 있으나 단독으로 행위할 수는 없는 파티">관찰자</abbr>(observer)**인지를 적는다.

- **서명자** — 그 컨트랙트의 생성·소비에 반드시 동의해야 하는 주된 권한자. "이 자산은 내 것"의 '내'.
- **관찰자** — 동의권은 없지만 그 컨트랙트를 **볼** 수 있는 파티.

A가 자산을 쓸 자격은 곧 **"그 자산 컨트랙트의 서명자가 A다"**라는 사실이다. 따로 잔액 테이블을 조회하는 게 아니라, 컨트랙트에 박힌 권한 그 자체다.

### 친숙한 것 → Canton 대응
| 친숙한 개념 | Canton |
|---|---|
| 사용자 계정 ID | 파티 ID |
| 로그인 비밀번호 | 서명 키(개인키) |
| "이 행을 수정할 권한" | signatory / observer / controller |
| 잔액 테이블의 owner 컬럼 | 컨트랙트의 signatory 필드 |

### 이더리움 비교
| | 이더리움 | Canton |
|---|---|---|
| 식별자 | 주소(키에서 파생, 익명) | 파티 ID `hint::키지문`(신원 가능) |
| 키 <abbr class="gloss" title="컨트랙트를 소비해 비활성으로 만드는 것(archive). 보관된 컨트랙트는 더 이상 쓸 수 없음">보관</abbr> | 항상 사용자(지갑)가 직접 | **<abbr class="gloss" title="키를 노드(밸리데이터)가 보관·서명하는 파티(=수탁형). '로컬'은 노드 시점 작명 — 키·신원이 노드 안에 있음">로컬 파티</abbr>**=노드가 보관 / **<abbr class="gloss" title="키를 파티 주인이 직접 보관하고 거래마다 외부 서명하는 파티(=자기수탁). '외부'는 노드 시점 — 키가 노드 밖에 있음">외부 파티</abbr>**=본인이 직접 |
| 행위자 | `msg.sender` 하나 | 한 트랜잭션에 **여러 서명자** 가능 |
| 가시성 | 모두 공개 | 서명자·관찰자만 |

가장 큰 차이는 **다자 권한**이다. 이더리움 호출의 행위자는 `msg.sender` 한 명. Canton 컨트랙트는 여러 파티가 함께 서명자가 될 수 있어, "양쪽이 동의해야 성립"하는 워크플로가 **네이티브**다(S6 정산의 토대).

## 심화

### 파티 ID는 실제로 어떻게 생겼나?
파티 ID는 두 부분이 `::`로 붙은 문자열이다.

```
app_user_<...>::1220<64자리 16진 키지문>
└── hint ──┘   └──── 네임스페이스(키 지문) ────┘
```

- **hint** — 사람이 읽는 힌트. 데모에선 참여자 종류에 따라 `app_user_…`(기관 A 쪽), `app_provider_…`(기관 B 쪽), `DSO::…`(자산 발행자), <abbr class="gloss" title="정산에서 주문을 매칭하고 원자적 실행을 개시하는 중립 당사자(venue). 자산을 보관하진 않음">운영사</abbr>는 힌트 `musubi-venue`로 만든다.
- **`::`** 뒤 **<abbr class="gloss" title="파티 ID의 `::` 뒤쪽 키 지문 부분. 그 파티를 만든 키(소유 주체)를 가리킴">네임스페이스</abbr>** — 그 파티를 만든 **키의 지문(fingerprint)**. `1220`은 멀티해시 접두(앞 2바이트가 해시 알고리즘·길이), 그 뒤가 공개키 해시. 즉 파티 ID에 **소유 주체의 키가 박혀 있다.** 이더리움 주소가 키에서 파생되는 것과 같은 정신이되, 사람이 읽을 힌트가 앞에 붙는다.

> 이 코스는 형식만 보여준다. `1220abcd…` 뒤 실제 지문 값은 노드·환경마다 다르다.

### 로컬 파티 vs 외부 파티 — 키를 누가 쥐나?
파티의 **키를 어디에 두느냐**가 두 유형을 가른다.

- **로컬 파티(local party)** — 키를 **<abbr class="gloss" title="파티를 호스팅하고 그 파티의 컨트랙트를 저장·실행하는 노드. 밸리데이터의 핵심 구성요소">참여자 노드</abbr>가 보관**한다. 노드가 그 파티를 대신해 서명한다. 운영이 단순(노드에 맡김). 데모의 기관 파티들이 이 방식.
- **외부 파티(external party)** — 키를 **파티 본인이 직접** 쥔다(노드 밖). 서명도 본인이 한다. 노드가 키를 못 봐서 **자기수탁(self-custody)**에 가깝다. 노드 운영자를 신뢰하지 않아도 되는 대신 키 관리 책임이 본인에게 온다.

이더리움에 빗대면, 외부 파티는 "내 지갑에 내 키"(<abbr class="gloss" title="이더리움에서 개인키로 직접 제어되는 외부 소유 계정(Externally Owned Account)">EOA</abbr>처럼), 로컬 파티는 "거래소가 내 키를 대신 보관"에 가깝다.

### 데모에서 파티를 고르는 코드 감각
백엔드는 노드에 "파티 목록"을 물어 hint 접두로 역할을 가려낸다. 개념적으로:

```python
# 참여자(노드)가 호스팅하는 파티 목록에서 역할별로 고른다
A   = next(p for p in parties if p.startswith("app_user_")     and p["isLocal"])
B   = next(p for p in parties if p.startswith("app_provider_"))
DSO = next(p for p in parties if p.startswith("DSO::"))        # 자산 발행자
```

`isLocal`이 참이면 그 파티의 키를 이 노드가 보관(로컬 파티)한다는 뜻이다.

## 강의 노트
- **핵심 한 문장**: "이더리움의 '주소'가 Canton에선 '파티'다. 단, 키를 노드가 쥘 수도(로컬) 본인이 쥘 수도(외부) 있고, 한 거래에 서명자가 여럿일 수 있다."
- **비유**: 파티 ID = '이름표(hint) + 지문(키)'. 이름표는 읽기 편하라고, 지문은 위조 못 하라고.
- **무엇을 보여주며 짚을지**: `app_user_…::1220…` 문자열을 실제로 보여주고 `::` 앞뒤를 손가락으로 가른다.
- **예상 질문 & 답**:
  - Q: "주소처럼 짧지 않고 왜 이렇게 긴가요?" → A: "신원(hint)과 키(지문)를 한 문자열에 담아서. 익명 주소가 아니라 누가 만든 파티인지가 드러난다."
  - Q: "한 기관이 파티 하나인가요?" → A: "설계에 따라. 유저마다 파티를 팔 수도, 소수 파티로 묶을 수도 있다(파티 생성 비용 때문에 후자가 흔함)."
