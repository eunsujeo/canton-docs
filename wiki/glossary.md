# 용어집 (Glossary) — 번역 일관성 규칙

모든 페이지 번역 시 이 표를 따른다. 새 용어를 만나면 여기에 먼저 추가하고 번역한다.

## 번역 원칙
- **고유 제품/프로토콜명·핵심 구성요소명은 원문 유지**: Canton, Daml, Splice, Canton Coin, Synchronizer, Global Synchronizer 등. (Synchronizer는 시퀀서·미디에이터 등 형제 구성요소와 톤을 맞추기 위해 영문 그대로 쓴다.)
- **기술 용어는 "한국어(영문)" 병기**를 첫 등장 시 1회**: 예) 재할당(reassignment). 이후에는 한국어만 사용.
- **코드, 명령어, 식별자, 파일 경로, API 이름은 절대 번역하지 않음.**
- 의미가 왜곡될 수 있으면 원문 용어를 그대로 두고 괄호로 설명을 단다.

## 툴팁 정의 (tooltip terms)
본문에서 이 용어가 처음 나올 때 `gen_tooltips.py`가 자동으로 `<abbr>` 툴팁을 붙인다.
**용어**는 본문에 나타나는 한국어 표기와 정확히 일치해야 하며, **정의**는 한 줄로 간결하게 쓴다.
새 페이지에서 설명이 필요한 용어를 만나면 먼저 이 표에 추가한다.

| 용어 | 정의 |
|---|---|
| Scan | 네트워크의 공개 통계·활동을 보여주는 익스플로러(블록 익스플로러의 Canton판) |
| Amulet | Canton Coin(CC)의 Daml/Scan상 기술적 이름. CC = Amulet |
| 마이닝 라운드 | CC가 발행·정산되는 시간 단위. 열림→발행중→닫힘으로 진행되며 라운드마다 기여 비례 보상 |
| Tap | DevNet·LocalNet에서 테스트용 CC를 무료로 받는 동작(수도꼭지/faucet). MainNet엔 없음 |
| 부분 트랜잭션 프라이버시 | 한 트랜잭션을 "뷰"로 분해해, 각 파티가 자신과 관련된 부분만 보도록 하는 Canton의 핵심 프라이버시 방식 |
| 글로벌 Synchronizer | 슈퍼 밸리데이터들이 공동 운영하는 Canton의 퍼블릭 조율(합의) 계층 |
| 슈퍼 밸리데이터 | 글로벌 Synchronizer를 운영하고 네트워크 거버넌스에 참여하는 노드 |
| Synchronizer | 상태를 저장하지 않고 트랜잭션 합의·순서를 조율하는 Canton 구성요소 |
| 밸리데이터 | 파티를 호스팅하고 그 파티의 컨트랙트 데이터를 저장하는 참여자 노드 |
| 파티 | Canton에서 권한과 데이터 가시성의 주체가 되는 식별 가능한 참여 주체 |
| 컨트랙트 | 원장에 기록되는 불변 데이터 단위. 상태 변경은 새 컨트랙트 생성으로 표현됨 |
| 템플릿 | 컨트랙트의 구조와 규칙(권한·초이스)을 정의하는 Daml 청사진 |
| 초이스 | 컨트랙트에서 수행 가능한 동작(권한이 부여된 당사자만 실행 가능) |
| 소비형 초이스 | 실행하면 그 컨트랙트를 보관(소비)하는 초이스. 상태 변경·이전에 쓴다(기본값) |
| 비소비형 초이스 | 실행해도 컨트랙트를 활성으로 남겨두는 초이스. 조회·알림·읽기에 쓴다 |
| 로컬 파티 | 키를 노드(밸리데이터)가 보관·서명하는 파티(=수탁형). '로컬'은 노드 시점 작명 — 키·신원이 노드 안에 있음 |
| 외부 파티 | 키를 파티 주인이 직접 보관하고 거래마다 외부 서명하는 파티(=자기수탁). '외부'는 노드 시점 — 키가 노드 밖에 있음 |
| 이해관계자 | 어떤 컨트랙트와 관계를 맺어 그것을 보거나 승인하는 파티 = 서명자 + 관찰자 |
| 서명자 | 컨트랙트의 주된 권한자. 생성·보관(소비)에 반드시 동의해야 하는 파티 |
| 관찰자 | 컨트랙트를 볼 수 있으나 단독으로 행위할 수는 없는 파티 |
| 뷰 | 한 트랜잭션을 당사자별로 나눈 조각. 각 당사자는 자기 권한에 해당하는 뷰(자기 몫)만 받아 본다 |
| 컨트롤러 | 컨트랙트의 특정 초이스(동작)를 실행할 권한을 가진 파티 |
| 원자성 | 트랜잭션이 전부 적용되거나 전혀 적용되지 않는 성질. 일부만 반영되는 일이 없음 |
| Daml | 다자간 워크플로를 위해 설계된 Canton의 스마트 컨트랙트 언어 |
| Canton Coin | 트랜잭션 수수료와 밸리데이터 보상에 쓰이는 네이티브 유틸리티 토큰(CC) |
| 토폴로지 | 어떤 노드·파티·키가 네트워크에 참여하는지를 정의하는 구성 정보 |
| 재할당 | 컨트랙트를 한 Synchronizer에서 다른 Synchronizer로 옮기는 프로토콜 |
| 프루닝 | 더 이상 필요 없는 과거 원장 데이터를 정리해 저장공간을 줄이는 작업 |
| 레이어 0 | 여러 L1 블록체인을 구축·연결하도록 받쳐주는 기반 인프라 계층(예: Polkadot, Cosmos) |
| 레이어 1 | 자체 합의로 트랜잭션을 직접 확정하는 기반 블록체인(L1). 다른 체인에 의존하지 않음 |
| 레이어 2 | L1 위에 얹혀 처리량을 늘리는 확장 계층(L2). 보안은 L1에 의존(예: Base, Arbitrum) |
| ZK-롤업 | 영지식 증명으로 다수 트랜잭션을 체인 밖에서 처리하고 유효성 증명만 L1에 올리는 L2 기술. 내용을 공개하지 않고 검증 가능해 프라이버시에도 활용 |
| 프라이빗 채널 | 특정 참여자끼리만 거래 데이터를 공유하는 별도 통로. 채널 밖에서는 내용이 보이지 않음(예: 상태/결제 채널, Hyperledger Fabric channel) |
| 트랜잭션 | 원장 상태를 바꾸는 원자적 작업 단위. 하나 이상의 컨트랙트를 생성·보관하며, 전부 적용되거나 전혀 적용되지 않음 |
| 스마트 컨트랙트 | 원장 위에서 규칙대로 자동 실행되는 코드화된 계약. Canton에선 Daml 템플릿으로 작성 |
| 참여자 노드 | 파티를 호스팅하고 그 파티의 컨트랙트를 저장·실행하는 노드. 밸리데이터의 핵심 구성요소 |
| 호스팅 | 참여자 노드가 파티를 대신해 원장에서 활동(컨트랙트 저장·트랜잭션 제출·확인)해 주는 것. 로컬 파티는 키까지 노드가 관리하고, 외부 파티는 제출 키를 파티 자신이 보유(노드는 중계) |
| 원장 | 거래·컨트랙트가 기록되는 장부. Canton에선 활성 컨트랙트의 모음 |
| 합의 | 여러 노드가 트랜잭션의 유효성·순서에 함께 동의하는 절차 |
| BFT | 비잔틴 장애 허용(Byzantine Fault Tolerance). 일부 노드가 악의적이거나 고장 나도 시스템이 올바르게 동작하는 성질 |
| ISS | Insanely Scalable State-machine replication. Canton 시퀀서가 메시지 전체 순서에 합의하는 데 쓰는 BFT 합의 알고리즘(1/3 미만 비잔틴 허용) |
| 시퀀서 | Synchronizer 구성요소. 암호화된 메시지에 전체 순서·타임스탬프를 부여하고 참여자에게 전달 |
| 미디에이터 | Synchronizer 구성요소. 이해관계자들의 확인을 모아 트랜잭션 승인/거부를 판정 |
| 커밋 | 트랜잭션이 최종 확정되어 원장에 반영되는 것 |
| 확인 | 이해관계자 밸리데이터가 트랜잭션이 유효함을 미디에이터에 응답하는 것(confirmation) |
| 활성 컨트랙트 | 아직 보관(소비)되지 않아 현재 유효한 컨트랙트 |
| 보관 | 컨트랙트를 소비해 비활성으로 만드는 것(archive). 보관된 컨트랙트는 더 이상 쓸 수 없음 |
| eUTXO | 확장 UTXO. 금액만이 아니라 임의의 상태·규칙을 담는 컨트랙트로 원장을 구성하는 모델 |
| 트래픽 | Synchronizer에 쓰기를 요청할 때 소비하는 자원. Canton Coin으로 비용을 지불 |
| DvP | 인도-대-지급(Delivery vs Payment). 자산 인도와 대금 지급을 동시·원자적으로 처리 |
| 토큰화 | 실물·금융 자산을 원장 위의 토큰(컨트랙트)으로 표현하는 것 |
| DSO | 탈중앙 Synchronizer 운영(Decentralized Synchronizer Operations) 파티. 슈퍼 밸리데이터들의 공동 거버넌스 주체 |
| CIP | Canton 개선 제안(Canton Improvement Proposal). 네트워크 규칙·표준 변경을 제안·비준하는 절차 |
| Splice | 글로벌 Synchronizer를 구동하는 오픈소스 애플리케이션 모음(SV·밸리데이터·월렛 등) |
| 커맨드 | 애플리케이션이 원장에 제출하는 명령(컨트랙트 생성·초이스 실행 요청) |
| 다자간 워크플로 | 여러 조직·당사자가 함께 참여하는 업무 흐름(예: 결제·정산·대출) |
| 이중지불 | 같은 자산을 두 번 쓰는 부정행위 |
| 원장 샤드 | 전체 원장 중 그 참여자 노드가 보관하는 자기 조각. 자기 파티가 이해관계자인 컨트랙트만 담김(전체 사본은 어디에도 없음) |
| 샤드 | 큰 데이터를 나눈 조각. Canton에선 참여자 노드가 보관하는 원장의 자기 몫 |
| 온-원장 | 원장(Daml 컨트랙트) 위에서 실행·기록되는 것. 모든 이해관계자가 공유·검증·강제 |
| 오프-원장 | 원장 밖, 내 백엔드 인프라에서 실행되는 것. 외부 API·UI·복잡 계산 등 나만 처리 |
| 확장성 트릴레마 | 탈중앙화·보안·확장성 셋을 동시에 만족하기 어렵다는 블록체인 설계 난제(보통 둘만 취함) |
| 데이터 가용성 | 트랜잭션 데이터가 검증자들이 실제로 받아볼 수 있게 공개·보관되는 성질(없으면 검증 불가) |
| 모놀리식 블록체인 | 실행·정산·합의·데이터가용성을 한 체인이 모두 처리하는 구조(예: Bitcoin, 초기 Ethereum) |
| 모듈러 블록체인 | 실행·정산·합의·데이터가용성을 계층으로 분리해 각자 최적화하는 구조(예: 롤업+Celestia) |
| 확정성 | 트랜잭션이 되돌려지지 않는다고 보장되는 상태. 확률적(점점 굳음) vs 결정적(즉시 최종) |
| 사이드체인 | 자체 합의를 가진 독립 체인을 브릿지로 메인체인에 연결한 것(보안을 빌리는 L2와 다름) |
